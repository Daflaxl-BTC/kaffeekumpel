## M3 — Umsetzung im Detail

### Datenfluss (neu)

```
Browser mit kk_session-Cookie
  │
  │ GET /api/realtime-token
  ▼
Next.js Route (Node runtime)
  ├─ readSessionCookie() → { group_id, member_id, slug, name }
  ├─ SignJWT (HS256, SUPABASE_JWT_SECRET)
  │     payload: { aud: "authenticated", role: "authenticated",
  │                sub: member_id, iss: "supabase",
  │                exp: now + 3600,
  │                app_metadata: { group_ids: [group_id] } }
  └─ { token } als JSON
  │
  ▼
Browser: supabaseBrowser().realtime.setAuth(token); channel.subscribe()
  │
  ▼
Supabase Realtime verifiziert JWT → setzt role=authenticated + claims
  │
  ▼
Policy "events scoped by jwt group_ids":
  group_id::text = any( jsonb_array_elements_text(auth.jwt()->'app_metadata'->'group_ids') )
```

### Rate-Limit

`/api/realtime-token` rate-limited auf 6 req/min pro IP (in-memory-Bucket pro
Lambda-Instanz). Reicht gegen versehentliche Loops im Hook; nicht gegen
verteilten Missbrauch gedacht — dafür müsste das Limit in Upstash o.ä. landen.

### Token-Refresh

Hook refresht Token alle 50 Minuten (Token-TTL 60 Minuten → 10 Minuten Puffer).
Keine harte Revocation-Liste: ein geklautes Token läuft schlimmstenfalls
1 Stunde, das Session-Cookie selbst ist davon unabhängig (SESSION_SECRET-signiert).

### Keine Mass-Logout-Gefahr

Existierende Session-Cookies bleiben gültig (eigener Schlüsselkreis). Nur der
Realtime-Pfad bekommt ab Merge einen zusätzlichen kurzlebigen Token — wer
bereits drin ist, wird durch die normale Navigation transparent rehydriert,
sobald der Hook das erste Mal läuft.

### Nicht geändert

- Server-Actions & RSC-Reads: laufen weiter über `SUPABASE_SERVICE_ROLE_KEY`
  und bypassen RLS.
- `supabaseBrowser()` bekommt weiter den anon-Key beim Setup; nur der
  Realtime-Channel wird über `setAuth` hochgestuft.
- Andere Tabellen (groups, products, members_public, settlements, debts) —
  deren Read-Policies sind nicht-PII und bleiben `using(true)`. Realtime ist
  nur auf `events` in der Publication.

## Test-Workflow (Supabase Pro)

Branching ist seit dem Pro-Upgrade möglich — Audit-Fixes gehen nicht mehr
direkt auf prod.

1. `create_branch` via MCP (~$0.013/h, kurzlebig).
2. Migration `20260419090000_security_audit_events_jwt_scoped.sql` applizieren.
3. Policy mit simuliertem JWT testen:
   ```sql
   set local role authenticated;
   set local request.jwt.claims = '{"app_metadata":{"group_ids":["<gruppe-a-id>"]}}'::jsonb;
   select count(*) from public.events where group_id = '<gruppe-a-id>';  -- > 0
   select count(*) from public.events where group_id = '<gruppe-b-id>';  -- = 0
   ```
4. Ohne Claims → 0 Rows:
   ```sql
   set local request.jwt.claims = '{}'::jsonb;
   select count(*) from public.events;  -- = 0
   ```
5. `merge_branch` auf prod, Branch danach löschen.
6. `get_advisors` → muss `[]` bleiben.

**Hinweis:** Erste Version der Policy nutzte `auth.jwt()` direkt — Performance-
Advisor hat `auth_rls_initplan` geworfen (pro-Row-Evaluation). Fix in
Follow-up-Migration `20260419091000_events_policy_rls_initplan_fix.sql`:
`(select auth.jwt())` statt `auth.jwt()`. Verhalten identisch, Postgres
evaluiert den Claim einmal pro Query.

## Acceptance (verifiziert bei Merge)

- [x] Ohne Session-Cookie: `fetch('/api/realtime-token')` → 401, Realtime bleibt offline.
- [x] Mit Session Gruppe A: nur events der Gruppe A werden gepusht.
- [x] Mit Session Gruppe A + manipuliertes Client-Filter auf Gruppe B: Postgres
      liefert keine Rows, Channel bleibt leise.
- [x] Kaffee tracken, Settlement finalisieren, Join, Share → unverändert.
- [x] Supabase Advisor post-merge: Security `[]`, Performance keine neuen
      WARN/ERROR-Lints (verbleibende INFOs `unused_index` /
      `auth_db_connections_absolute` sind pre-existing, nicht durch M3
      verursacht).
